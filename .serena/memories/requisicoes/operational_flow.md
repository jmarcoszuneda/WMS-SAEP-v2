# Requisicoes operational flow
- Last verified: 2026-05-26 from `main` at merge PR #27 (`5cf299c`). Flow #19-#27 is implemented in code and docs.
- State machine uses `EstadoRequisicao`: `rascunho`, `aguardando_autorizacao`, `recusada`, `autorizada`, `pronta_para_retirada`, `atendida`, `cancelada`, `estornada`.
- Main service entry points live in `apps/requisicoes/services.py`: `criar_requisicao`, `editar_rascunho`, `enviar_para_autorizacao`, `retornar_para_rascunho`, `recusar_requisicao`, `autorizar_requisicao`, `separar_para_retirada`, `registrar_atendimento`, `cancelar_ou_descartar_requisicao`, `cancelar_requisicao`.
- Query/read side lives in `apps/requisicoes/selectors.py`: `materiais_para_requisicao`, `requisicoes_visiveis_para`, `minhas_requisicoes`, `fila_autorizacao`, `fila_atendimento`.
- Permission rules live in `apps/requisicoes/policies.py`: creation scope/beneficiary checks, draft edit/send, authorization queue and actions, fulfillment queue/actions, cancel before pickup. Superuser bypass exists for operational actions; inactive users denied.
- Views live in `apps/requisicoes/views.py`; templates live under `apps/requisicoes/templates/requisicoes/`: operational landing, my requisitions, detail/timeline, draft form with HTMX item rows/material autocomplete, authorization queue, fulfillment queue, fulfill form.
- Draft flow: create/edit require active actor, allowed beneficiary scope, at least one valid material item, positive quantity. First send generates public number `REQ-AAAA-NNNNNN`; returned draft preserves number.
- Visibility: creator sees own drafts; beneficiary sees non-drafts; sector chief sees own sector except third-party drafts; almoxarifado sees all non-drafts; superuser sees all.
- Authorization: `fila_autorizacao` shows `aguardando_autorizacao` for chief of beneficiary sector. Chief can return to draft, refuse with required reason, or authorize integral only. Authorization reserves stock, sets each item authorized equal to requested, and does not lower physical stock.
- Fulfillment: `fila_atendimento` shows `autorizada` and `pronta_para_retirada`. Almoxarifado separates `autorizada -> pronta_para_retirada` without stock movement, then registers total/partial fulfillment from `pronta_para_retirada -> atendida`.
- Fulfillment quantities: delivered must be > 0 overall, <= authorized per item, reject NaN/sNaN, require per-item justification when delivered < authorized or zero, lower physical stock only by delivered quantity, consume delivered reservation, release non-delivered reservation.
- Cancel before pickup: cancel/descartar service handles rascunho never sent as discard, numbered draft or awaiting authorization as logical cancel without mandatory reason, authorized/pronta as cancel with mandatory justification and reservation release, no physical stock change. Final states block cancel.
- Concurrency rule from docs still applies: transition services use `transaction.atomic`, lock `Requisicao` with `select_for_update`, then lock stock rows in deterministic order when stock/reservation changes.
